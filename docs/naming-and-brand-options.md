# Naming and Brand Options

## Current state

"Mole" is descriptive and useful, but not especially punchy or ownable as an open-source product name.

## Recommended current posture

Keep:
- repository/product model name: **Mole**
- command/tool name: **cascade**

This gives the project a more usable product surface without forcing a rushed rename.

## Why this is sensible

- the descriptive name still helps people understand what it is
- the CLI name is short and memorable
- open-source users can adopt the command vocabulary early
- a stronger full rebrand can happen later if it becomes obviously worth it

## Future full-brand directions

Potential directions to explore later:
- names about flow, signal, synthesis, or context
- names that feel like tools rather than frameworks
- names that work as both repo name and CLI/package name

## Recommendation

Do not spend too much energy naming too early.
Make the command surface good first.
If the product starts to resonate, the right stronger name will be easier to recognise.
